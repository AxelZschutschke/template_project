23 ## Introduction
24 
25 The test plan is based on the [test strategy](\ref TestStrategy), which it
26 translates into concrete test scenarios and time lines. While the test strategy
27 usually is changing only slightly from release to release (representing the
28 learning process of the team and changing requirements and environments during
29 developement), the test plan usually changes significantly per release, as
30 new requirements are implemented or old requirements are signed off.
31 
32 ## Inspection Specifications / Procedure
33 
34 There are two major reasons for carrying out the software test:
35 * test if newly introduced functionality is working as expected
36 * test for absence of regression: old functionality shall not be broken by new features
37 
38 Automated fast tests and analysis tools generate reports per build, which are thus always
39 up to date. Tests may be executed on a slower interval as well, e.g. manual tests or
40 weekly (long-running) system tests. The test results of these less-frequent tests are
41 valid only for the exact build they have been carried out with. Especially for manual
42 run test cases, an exact documentation which versions and environment have been engaged
43 is necessary.
44 
45 ### Defect Handling
46 
47 Minor defects found during software test should be corrected straight away,
48 for major or complex-to-solve issues an bug tracking item shall be created.
49 The reason for this is twofold:
50 * make the problem visible to everyone as fast as possible (even if fixing is taking some time)
51 * enable a discussion on how to solve (or document if fixture is not realistic) the issue

53 # Test Description
54 ## Unit Test
55 Unit tests shall be implemented for each individual component. The tests themselves should be classified
56 as black box test. This implies, that a tester only needs information about the unit interfaces and does
57 not need to know about the inner structure of the system under test.
58 The tester may be the same person as the developer, as unit tests shall proof that the method provides
59 the desired functionality and does not show unexpected behavior. For implementing the unit test a set
60 of style guides is formulated.
61 
62 ### Style Guide
63 
64 1. naming of source files for test:
65  <unit_name>Test.cpp e.g. JSONParser.cpp
66 2. creating stubs/mocks/fakes
67  <unit_name>Fake.h e.g. JSONParserFake.h
68 3. test classes / suites are named by their class under test:
69  <unit_name>Test<_specific_aspect> e.g. JSONParserTest_Strings
70 4. a test name is assembled like
71  <scenario>_<expectedBehavior> e.g. whenValidStringEntered_shallConstructJSONStringObject
72 5. camel cases for testnames start lowerCase
73 6. camel cases for any class names start UpperCase
74 7. tests do not contain logic (if, for, while, switch are bad)
75 8. tests only test agains public methods containing logic, esp. simple setters/getters are not tested!
76 9. tests shall be consistent (no random, time- or hardware dependend data shall be used for checking)
77 10. tests shall be independend of each other, e.g. shall not change the state of the test system
78 11. use the arrange-act-assert pattern and dont mix up asserts with method calls
79 12. for test code, refer to the code style guide
80 13. test code shall be commented with care

86 ## Explorative Testing
87 
88 Explorative testing is a special kind of manual testing without a pre-described test proceduce. It has
89 been found highly advantageous. A tester sets up a test-environment as close to production as possible
90 and documents all hardware and software components in the environment. Then the tester starts a timeboxed
91 test, mostly 90 minutes, where the tester can perform any operations on the system and records the actions
92 and observations. If the observation is not equal to the expectation, the test is accounted for as failure.
93 
94 It makes sense to set up the mood of the tester to either "intruder" or "developer" for the test, so the
95 tester either tries to prove that the system works as expected or tries to break the system intentionally.
96 
97 ## Code Coverage
98 
99 The unit tests are run multiple times with different build options. One build is carried out with code
100 instrumentation for measuring code coverage during unit test. This measure describes how much code has
101 been executed in total during the unit test. A project should strive for 100 % code (line) coverage and
102 a high branch coverage (as high as 100 % for security or mission critical code, see
103 [ASIL](https://en.wikipedia.org/wiki/Automotive_Safety_Integrity_Level)). Here the following coverage
104 levels are set as target:
105 
106 * line coverage: 100 % ( > 95 % is green, > 80 % is yellow and less is red )
107 * branch coverage: 80 % ( > 75 % is green, > 60 % is yellow and less is red )
108 
109 The reports are generated using gcc, gcov and lcov for compiling the coverage reports.
110 
111 Combining unit tests and test coverage metrics blurrs the boundary of black box and white box testing.
112 This is not a problem in general, as the tester may be informed about whitespots in the test by this
113 method. Care should be taken, as untrained testers tend to "only increase the coverage" while not
114 creating strict tests anymore.
115 
116 ## Static Code Analysis
117 
118 Static code analysis is used as a first step of code review, not replacing the code review of other
119 developers and testers. It assists the developer and the reviewer for standard tasks (e.g. are all
120 class members initialized?) and simple declare / define / use checks. Static code analysis is classified
121 as white-box test/
122 
123 Here, cppcheck is utilized for this. The tool creates an xml report for the source base. Unfortunately,
124 the tool has some issues with header-only libraries. Thus the result must be handled with care. The code
125 coverage of the unit test also forces the cppcheck tool to really examine all code of the header only
126 parts of the library.
127 
128 Supressions may be introduced with care and shall be reviewed for each release.
129 
130 ## Memory Check
131 
132 The tool valgrind/memcheck is utilized for a special unit test configuration and execution. It creates
133 an jailed environment and replaces standard libraries with its own tracing tools. It is a powerfull tool
134 for detecting unitialized variables and memory leaks as well as buffer overflows.
135 
136 Supressions may be introduced with care and shall be reviewed for each release.
137 
138 ## Multi-Thread Check
139 
140 The tool valgrind/helgrind is utilized for a special unit test configuration and execution. It creates
141 a jailed environment and replaces standard libraries with its own tracing tools. It is a powerfull tool
142 for detecting potential race conditions, dead locks or unprotected thread interactions.
143 
144 Supressions may be introduced with care and shall be reviewed for each release.
145 
146 ## Code Metrics
147 
148 The tool CCCC is utilized to generate a report about overall code metrics, e.g. lines of code, cyclomatic
149 complexity, lines of comment / lines of code etc. These numbers, while beeing less expressive if viewed in
150 isolation, might help during refactoring and review. For example could a refactoring reduce the complexity
151 of the code and increase the number of comments.
152 The numbers are useful for comparing source files and projects (of the same language and standard)
153 against each other.
