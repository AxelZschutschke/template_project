7 Describes the general rules for styling and outlining code in this project. Describes the naming conventions for
8 variables, classes, namespaces etc.

12 ## Tooling
13 
14 To enforce a common appearance of the codebase, the tool clang-format is utilized, which describes indentation
15 and define patterns in classes. Unfortunately, it cannot protect from mis-styled variable names etc.

17 # Rules
18 
19 ## Files
20 
21 Files are named according to their contents either `*.h` for header files, `*.cpp` for implementation etc. Filenames are
22 started with a capital letter and continued camel case. Interfaces get an capital `I` as prefix, e.g. `IParser.h`
23 
24 ## Namespaces
25 
26 Namespaces are used for each sub-module individually and may be nested. A special namespace is Test, encapsulating
27 all test related definitions and test cases.
28 
29 Namespace names are started with lower case letter and continued in camel case like: `myLongNamespace`
30 
31 Namespaces should be documented briefly, to do this only once, please create (if not existing) a header file with
32 the same name (e.g. Namespace.h), declaring and document the namespace in there.
33 
34 ~~~~~.cpp
35 namespace {
36 }
37 ~~~~~
38 
39 ## Classes
40 
41 Like namespaces, start with capital letter and continue camel case.
42 
43 If classes serve as *pure virtual* interface, they get prefixed by an capital `I`, e.g. `class IParser`, while the
44 implementation does not receive an prefix, e.g. `class Parser : public IParser`
45 
46 Define only one class per file and use the class name as the filename, e.g. the `class Parser` would be defined in the
47 file `Parser.h`. Classes should be nested in namespaces. Test files and test classes affilitated to the original are
48 named accordingly with a \*Test or \*Fake or \*Stub attached, e.g. the file `ParserTest.cpp` could contain the `class
49  ParserTest` and the respective test methods.
50 
51 ~~~~~.cpp
52 namespace {
53  class ClassName {
54  };
55 }
56 ~~~~~
57 
58 ## Methods
59 
60 Free functions and class member methods get an expressive name, which does not need to be short. A maximum of 3
61 parameters is used (while not strictly prohibiting more parameters in special cases) and output parameters are avoided.
62 
63 Ideally, a method does not create "side effects" and does only modify the system by its return values (as teached by
64 functional programming). Methods shall only do one thing (contain a single logic), that does not imply that they are
65 prohibited to combine other methods, as the logic may be exactly this combination (but then this is the only thing they
66 should be doing). The ideal length of a method is approx. 5 lines.
67 
68 Method names start with a lower case letter and continue with camel case (if the language does not dictate otherwise,
69 e.g. for constructors) . Underscores and other special characters shall not be used (if not mandatory, e.g. destructors
70 or operators). An exception for this rule are test names, which contain an underscore character to seperate the scenario
71 and expectation part.
72 
73 ~~~~~.cpp
74 namespace {
75  class ClassName {
76  ClassName()
77  {}
78  ~ClassName()
79  {}
80 
81  int method( int const & input ) const
82  {}
83  };
84 
85  int freeFunction( std::string const & input )
86  {}
87 }
88 ~~~~~

90 ## Statements
91 
92 Only one statement in terms of programming logic shall be used per line, such that line coverage = statement coverage.
