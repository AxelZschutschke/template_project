13 # Status
14 
15 The tamplate project contains document templates, giving an outline for the actual project's documentation. They should
16 be reviewed and modified to your needs at project start, they are merely serving as some guidance for not forgetting
17 about parts of the documentation.
18 
19 The template project itself is under development, it does not contain everything I think should be in here. There are
20 time constraints for not having it in better shape. Things missing:
21 
22 * example code - I would like to add some examples with intentional errors for several reasons:
23  - get some documentation output
24  - get some testreport output
25  - show how the open source tools work for the specific issues
26 
27 * more documentation
28  - writing down the definitions and test documentation helps a lot to sort out things for myself
29  - takes a lot of time to get it right
30  - would take even more time to get it nice with some pictures etc.
31 
32 * requirement engineering
33  - I did not find an open source tool for that which could be integrated here
34 
35 * test reporting
36  - I am currently working on an offline python-based "dashboarding" tool creating md files as docygen input
37  - might be a personal thing: I am not all in for online dashboards, helpful as they may be -> I miss the "complete
38  overview" in there and the long term documentation
39  - my preffered way: having it all in the code doc (doxygen), create your architecture, requirement and test
40  documentation in there and just store the one folder with doxygen results along with your artifacts and maybe a copy
41  of your sources. That way you can easily take up work from there even if your preffered online dashboard cannot show
42  you your year-old info anymore.
