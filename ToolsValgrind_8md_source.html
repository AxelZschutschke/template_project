5 # Overview
6 
7 Valgrind is not a single tool but contains a complete tool suite for detecting memory management and threading bugs. It
8 is developed by ["a loose knit group of developers"](https://valgrind.org/info/developers.html) and is available under
9 the GNU general public license V2.0 as open source software.
10 
11 ## Get Valgrind
12 
13 Get it from your OS's package store, most Linux distributions have valgrind available, e.g. via `apt install valgrind`.
14 
15 Alternatively, valgrind can be built from source using the [git repository](git://sourceware.org/git/valgrind.git)
16 
17 To build the cloned code, follow the instructions in the README file that the clone should give you. Alternatively, the following should work:
18 ~~~~~.bash
19 cd valgrind
20 ./autogen.sh
21 ./configure --prefix=...
22 make
23 make install
24 ~~~~~

26 ## Using Valgrind
27 
28 As mentioned before, valgrind contains a set of tools to examine the binaries in a dynamic manner. *dynamic* means, that
29 the code is actually executed for the test. Valgrind sets up a modified execution environment simulating certain CPU
30 features and memory. Thus, the execution via valgrind is considerably slower than the original executable and may
31 allocate more memory. For big projects, it may be beneficial to deconstruct the binary into smaller chunks for
32 examination.
33 
34 The individual tools are described in the following.
35 
36 ## Memcheck
37 
38 Memcheck, a tools for examining the memory allocation, deallocation and initialization process of the executable. It
39 finds uninitialized variables used e.g. for conditions, buffer overflows, memory leaks etc.
40 
41 To execute memcheck, issue the following command:
42 
43 ~~~~~.bash
44 valgrind --tool=memcheck --trace-children=yes --leak-check=full --track-origins=yes --suppressions=<file> <program>
45 ~~~~~
46 
47 Here, some options are passed to valgrind already, to follow child threads of the main process (`trace-children`), track
48 the origin of error and leaks (`track-origin`) and perform extended checks for memory leaks (`leak-check`). The option
49 `tool=memcheck` may be skipped, as it is the default option.
50 
51 Valgrind can use a suppressions file to suppress known issues and false-positives. This file can be created during an
52 initial execution with the option `--gen-suppressions=<file>`.
53 
54 Valgrind also supports an xml output, which can be parsed by different dashboard tools, e.g. by a jenkins plugin (which
55 is a little buggy, though).
56 
57 ## Cachegrind
58 
59 Cachegrind is a cache profiler. It performs detailed simulation of the I1, D1 and L2 caches in your CPU and so can
60 accurately pinpoint the sources of cache misses in your code. It identifies the number of cache misses, memory
61 references and instructions executed for each line of source code, with per-function, per-module and whole-program
62 summaries. It is useful with programs written in any language. Cachegrind runs programs about 20--100x slower than
63 normal.
64 
65 It is usable for very small examples only, as the caching behavior gets blurred by too many different memory requests.
66 
67 ~~~~~.bash
68 valgrind --tool=cachegrind --trace-children=yes <program>
69 ~~~~~
70 
71 ## Callgrind
72 
73 Callgrind, by Josef Weidendorfer, is an extension to Cachegrind. It provides all the information that Cachegrind does,
74 plus extra information about callgraphs. It was folded into the main Valgrind distribution in version 3.2.0. Available
75 separately is an amazing visualisation tool, **KCachegrind**, which gives a much better overview of the data that Callgrind
76 collects; it can also be used to visualise Cachegrind's output.
77 
78 ~~~~~.bash
79 valgrind --tool=callgrind --trace-children=yes --callgrind-out-file=<file> <program>
80 kcachegrind <file>
81 ~~~~~

83 ## Massiv
84 
85 Massif is a heap profiler. It performs detailed heap profiling by taking regular snapshots of a program's heap. It
86 produces a graph showing heap usage over time, including information about which parts of the program are responsible
87 for the most memory allocations. The graph is supplemented by a text or HTML file that includes more information for
88 determining where the most memory is being allocated. Massif runs programs about 20x slower than normal.
89 
90 **ms\_print** may be used to analyze the outfile of the massif tool (which is hard to read by humans).
91 
92 ~~~~~.bash
93 valgrind --tool=massif --trace-children=yes --massif-out-file=<file> <program>
94 ms_print <outfile>
95 ~~~~~

97 ## Helgrind
98 
99 Helgrind is a thread debugger which finds data races in multithreaded programs. It looks for memory locations which are
100 accessed by more than one (POSIX p-)thread, but for which no consistently used (pthread\_mutex_) lock can be found. Such
101 locations are indicative of missing synchronisation between threads, and could cause hard-to-find timing-dependent
102 problems. It is useful for any program that uses pthreads. It is a somewhat experimental tool, so your feedback is
103 especially welcome here.
104 
105 ~~~~~.bash
106 valgrind --tool=helgrind --trace-children=yes <program>
107 ~~~~~
